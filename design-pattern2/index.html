发表于 2018-10-01 | 分类于 编程思想

漫谈设计模式

工厂方法(Factory Method)模式

其实这个模式在暑假学习spring的时候有接触到，当时花了一阵子弄明白了，csdn_blog

GoF为工厂方法模式给出的定义如下

Define an interface for creating an object,but let subclasses decide which class to instantiate.Factory Method let a class defer instantiation to subclasses.

神图秒懂，我花了好久画的: )

比如，客户现在要买一个鼠标。鼠标有厂家吧

就是，你要有一个抽象工厂（是所有牌子的工厂的抽象），让这个抽象工厂的子类工厂去决定做什么品牌。

你要有一个抽象产品类吧（接口也行），因为比如无线鼠标，有线鼠标的构造也有点不一样，就由具体的产品去实现这个接口。

下面说一个比较任意搞混的，也叫static factory method。它不能算是一种模式吧，就是在你在创建对象的时候可以用的一种方法。

静态工厂方法

考虑使用静态工厂方法代替构造器

创建对象的时候，不再使用常规的new ，而是在newIntance(),getInstance()这些方法内部，创建对象（良好的封装，可以使用单例，也可以返回一个子类）

原型(Prototype)模式

比如，你家刚做了装修，你的朋友也希望给他家做装修，他可能会把你家的装修方案拿过去改改就成，你的装修方案就是原型。

GoF给出的原型模式定义为：

Specify the kinds of objects to create using a prototypical instance,and create new objects by copying this prototype.

简单地说，就是复制一个对象。

IOC

大量的使用工厂方法模式，会引起诸多问题。实际中，经常出现几百个服务类，使用工厂方法模式，可能出现几百个工厂类。这几百个Service对象之间往往存在复杂的依赖关系。

框架和工具库的区别就是：如果框架没有实现IOC，框架就退化为工具库。

装饰器模式

装饰模式为已有类动态附加额外的功能就像LOL、王者荣耀等类Dota游戏中，英雄升级一样。每次英雄升级都会附加一个额外技能点学习技能。具体的英雄就是ConcreteComponent，技能栏就是装饰器Decorator，每个技能就是ConcreteDecorator；

可能你直接看代码会比较费解，最好的方式是在IDEA下打好断点，一步步debug模式下看看程序是如何跳转的。

一个明显的缺点是：使用装饰模式会产生比使用继承关系更多的对象。更多的对象会使得查错变得困难，特别是这些对象看上去都很相像。

设计模式在JAVA I/O库中的应用

​ ●　　抽象构件(Component)角色：由InputStream扮演。这是一个抽象类，为各种子类型提供统一的接口。

　　●　　具体构件(ConcreteComponent)角色：由ByteArrayInputStream、FileInputStream、PipedInputStream、StringBufferInputStream等类扮演。它们实现了抽象构件角色所规定的接口。

　　●　　抽象装饰(Decorator)角色：由FilterInputStream扮演。它实现了InputStream所规定的接口。

　　●　　具体装饰(ConcreteDecorator)角色：由几个类扮演，分别是BufferedInputStream、DataInputStream以及两个不常用到的类LineNumberInputStream、PushbackInputStream。

观察上面的代码，会发现最里层是一个FileInputStream对象，然后把它传递给一个BufferedInputStream对象，经过BufferedInputStream处理，再把处理后的对象传递给了DataInputStream对象进行处理，这个过程其实就是装饰器的组装过程，FileInputStream对象相当于原始的被装饰的对象，而BufferedInputStream对象和DataInputStream对象则相当于装饰器。
